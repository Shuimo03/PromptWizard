# 通用 Prompt 模板

这些模板优先面向代码仓库协作场景，强调三件事：

- 先基于现有代码做判断，不凭空设计。
- 输出可执行结果，而不是泛泛建议。
- 明确约束、验收标准和受影响文件。

## 1. 实现类任务

~~~~markdown
# 角色
你是该仓库的工程师，需要先阅读现有实现，再动手修改。

# 目标
请实现以下需求：{{task}}

# 仓库上下文
- 工作目录：`{{cwd}}`
- 相关文件：
  - `{{file_1}}`
  - `{{file_2}}`
- 当前技术栈：{{stack}}

# 约束
- 不要凭空创建新的架构层，除非当前代码已经使用。
- 保持现有命名、导入方式和代码风格。
- 如果发现文档和代码不一致，以代码为准。

# 输出要求
- 直接给出需要修改的代码。
- 说明改动涉及哪些文件。
- 列出验证方式。
~~~~

## 2. 修复类任务

~~~~markdown
# 问题
{{bug_description}}

# 复现线索
- 入口：`{{entry_file}}`
- 相关模块：`{{related_file}}`
- 错误信息：{{error_message}}

# 任务
请先定位根因，再给出最小修复方案。

# 要求
- 解释根因。
- 避免顺手做无关重构。
- 如果需要新增测试，补充最小必要测试。
~~~~

## 3. 代码评审类任务

~~~~markdown
# 任务
请审查以下改动。

# 关注点
- 行为回归
- 边界条件
- 命名和目录是否与仓库一致
- 测试是否覆盖关键路径

# 输出格式
1. Findings: 按严重程度列问题。
2. Risks: 说明未覆盖但值得注意的点。
3. Summary: 用 2-3 句话概括。
~~~~

## 4. 文档修订类任务

~~~~markdown
# 任务
请修订以下文档，使其与当前仓库一致：`{{doc_path}}`

# 校验要求
- 路径必须真实存在。
- 依赖名、环境变量名、命令必须能在仓库里对上。
- 示例代码必须语法正确。
- Markdown 代码块必须可复制。
~~~~

## 5. Prompt 编写检查清单

- [ ] 是否引用了真实存在的路径和文件名
- [ ] 是否使用了当前仓库真实依赖
- [ ] 是否给出了明确输入、目标和约束
- [ ] 是否写清楚输出应该是代码、方案还是评审结论
- [ ] 是否提供了验证方式
- [ ] 是否避免了无效的嵌套代码块
